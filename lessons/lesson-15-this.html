1First, delete a lie

Almost every beginner is taught one sentence:

✕ The Myth

“this refers to the current object.”

Sometimes that's true. Sometimes it's completely false. A rule that's only sometimes right is worse than no rule — so throw it away. We're going to build the real one from the engine up.

◆ The key fact

this does not go through that chain. It is not a variable. The engine never searches the lexical environment for it. This single fact explains every confusing this you'll ever meet.

3this is born with the Execution Context

Remember: every time a function runs, the engine spins up an Execution Context. this is a slot inside that context — not inside the lexical environment.

The consequence

Because the context is created every time the function is called, this is decided at call time — not when the function was written. Variables are the opposite: fixed at creation. That difference is the whole lesson.

5The moment intuition breaks

predict before runningconst fn = naruto.speak;   // just grab the function
fn();                       // how was this called?

Most people say "Naruto" — because the function belongs to Naruto. But functions don't care where they're stored (you learned that with closures). And this doesn't care where the function was created. It asks one thing:

The only question

“Who called me?”

Not who owns me. Not who created me. The caller.

Here fn() was called as a plain function — no object. in front. There's no owner at the call site, so this becomes undefined (strict mode) — reading .name off it throws. We finish that thread in §8.

One function. Two call sites. naruto.speak() → this = naruto → "Naruto". sasuke.speak() → this = sasuke → "Sasuke". The dot's left-hand side becomes this.

8The last piece: fn() alone

When a function is called with nothing to the left of a dot, there's no caller-object. What fills the this slot then depends on strict mode:

Strict mode ("use strict", modules, classes)

this → undefined.
Reading .name off it throws a TypeError. Honest and loud.

Sloppy mode (old scripts)

this → the global object (window in browsers). Silent, surprising bugs.

✓ Rule of thumb

Modern code (ES modules, class bodies) is always strict. Assume a lone fn() gives this === undefined. If a bare call seems to “work,” you're probably in a sloppy old script.

9Your challenges — reason, don't run

Reason like the engine. Answer, then reveal.

Challenge 1 
naruto.speak();  // speak logs this.name

What prints, and what is this?

Called as object.method(). The engine sees the dot, puts the left-hand object (naruto) in the this slot. this.name → "Naruto".
Challenge 2 
const fn = naruto.speak;
fn();

What is this here?

Called as a plain fn() — no dot, no owner. In strict mode this = undefined, so undefined.name throws. Storage doesn't matter; the call site does.
Challenge 3 
const sasuke = { name: "Sasuke", speak: naruto.speak };
sasuke.speak();

What prints?

Same function object, but the call site is sasuke.speak(). Left of the dot = sasuke → this = sasuke → "Sasuke". Where it was defined is irrelevant.
Challenge 4 · the intermediate filter 
speak() { return this; }
// naruto.speak and sasuke.speak point to this same fn
console.log( naruto.speak() === sasuke.speak() );

true or false?

naruto.speak() returns the naruto object; sasuke.speak() returns the sasuke object. Two different objects → naruto === sasuke is false. The function is identical; the this each call returns is not. That's the whole lesson in one line. 🔥

✦What to carry forward

  • this is not a variable — it's a slot in the execution context.
  • It's set at call time, freshly, on every single call.
  • The rule is the call site, not the definition: “who called me?”
  • obj.method() → this = obj (left of the dot).
  • fn() alone → this = undefined in strict mode.

Yes — with call, apply, bind, and then the big twist: arrow functions, which throw the whole call-site rule out and capture this lexically (like a closure).
